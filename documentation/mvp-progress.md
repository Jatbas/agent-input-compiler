### Phase W — Global Server & Per-Project Isolation

Single global MCP server registered in `~/.cursor/mcp.json`. One server process handles all workspace folders. Each `aic_compile` call specifies its `projectRoot`; the server creates or reuses a `ProjectScope` per project via a `ScopeRegistry`. The database is global at `~/.aic/aic.sqlite` with a `project_root` column on per-project tables. Per-project files (`aic.config.json`, `.aic/project-id`, `.cursor/rules/AIC.mdc`, `.cursor/hooks/`) remain in each project directory. A stable UUIDv7 project ID in `.aic/project-id` survives folder renames. Duplicate detection warns when AIC is registered in both global and workspace configs. See `implementation-spec.md` §Phase W for full details.

## Phase 1.5 — Performance Optimizations

Minimize CPU, memory, and I/O usage of the MCP server at runtime. Every task below was validated against the current codebase — optimizations already in place (async file I/O, mtime-based file content caching, `WatchingRepoMapSupplier` file watcher, `SqliteCacheStore` compilation caching, lazy scan with bytes/4 estimate, fast-glob double-stat elimination, `isolatedModules`/`skipLibCheck` in tsconfig) are excluded. All changes are bounded to `mcp/src/` and `shared/src/` — zero pipeline algorithm changes.

### Phase X — Hot-Path I/O Elimination

Remove synchronous filesystem operations from the compile request path. Currently every `aic_compile` call triggers sync I/O for prompt log writing, project init checks, trigger rule verification, and hook installation — blocking the event loop and adding latency.

| Component                                             | Status | Package           | Deps | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ----------------------------------------------------- | ------ | ----------------- | ---- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| P01: Async prompt log write in compile handler        | Done   | mcp/src/handlers/ | —    | Replace `fs.writeFileSync(lastPromptPath, ...)` in `compile-handler.ts` line 140 with `fs.promises.writeFile`. The write is already wrapped in try/catch and marked non-fatal, so making it async has zero risk. Eliminates one blocking I/O call per compilation.                                                                                                                                                                                                                               |
| P02: Guard per-compile init/install behind once-flags | Done   | mcp/src/handlers/ | —    | `compile-handler.ts` calls `ensureProjectInit`, `reconcileProjectId`, `installTriggerRule`, and `installCursorHooks` on every `aic_compile` invocation (lines 86–95). Each performs multiple `fs.existsSync`/`fs.readFileSync`/`fs.writeFileSync` calls. Add per-`projectRoot` `Set<string>` guard so these run once per server lifetime per project, then skip. The guard resets on server restart, so correctness is preserved. Eliminates ~20 sync I/O calls per compilation after the first. |

### Phase Y — Object Allocation & Computation Reduction

Reduce per-request heap allocations and redundant computation in the compilation hot path. Fewer short-lived objects mean less GC pressure; cached intermediate results avoid repeated work.

| Component                                                 | Status | Package              | Deps | Description                                                                                                                                                                                                                                                                                                                                                                                                       |
| --------------------------------------------------------- | ------ | -------------------- | ---- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| P06: Replace object spread in `recordSessionStepIfNeeded` | Done   | shared/src/pipeline/ | —    | `compilation-runner.ts` line 218–219 uses `{ ...acc, [f.path]: f.tier }` inside a `reduce`, creating a new object on every iteration. Replace with `Object.fromEntries(r.prunedFiles.map(f => [f.path, f.tier]))` or mutable accumulation. For a 50-file compilation, this eliminates 49 intermediate object allocations.                                                                                         |
| P07: Cache `serializeRepoMap` hash per repo map instance  | Done   | shared/src/pipeline/ | —    | `serializeRepoMap()` in `compilation-runner.ts` creates a sorted copy of `repoMap.files` and serializes it on every `CompilationRunner.run()` call (line 307). Eliminates array sort + string construction on cache-hit compilations. |

### Phase Z — Memory Bounds

Cap unbounded in-memory data structures to prevent long-running server memory growth.

| Component                                       | Status | Package              | Deps | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ----------------------------------------------- | ------ | -------------------- | ---- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| P08: LRU bound on `CachingFileContentReader`    | Done   | shared/src/adapters/ | —    | The `Map` in `createCachingFileContentReader` (line 14) grows without bound — every file ever read is cached for the server's lifetime. For a 1000-file repo with an 8KB average, this is ~8MB of retained strings. Add a simple Map-based LRU with a configurable max size (default 500 entries). Ensures memory stays bounded during long sessions.                                                                                                                                                                                                                                                                                   |
| P09: Bounded `runnerCache` with watcher cleanup | Done   | mcp/src/             | —    | The `runnerCache` Map in `server.ts` (line 217) caches `CompilationRunner` per project root forever. Each runner holds `PipelineStepsDeps` including a `WatchingRepoMapSupplier` with an active `fs.watch` file watcher. Cap at 10 entries. On eviction, close the watcher to release the OS file handle. Use the already-exported `createPipelineDeps` (returns `PipelineDepsWithoutRepoMap`) instead of `createFullPipelineDeps` in `getRunner`, create `WatchingRepoMapSupplier` manually, and store `{ runner, closeable }` in the cache. Notes:

- **Dependency order:** P01/P02 are done. P06/P07 are independent of each other. P08/P09 are independent of each other and of all prior tasks.
- **Measurement:** Before and after each task, measure compilation latency via `aic://last` `durationMs`, and server RSS via `process.memoryUsage().rss` after 10 consecutive compilations. Log results in the daily log.
- **No algorithm changes:** All tasks modify infrastructure/plumbing only. Pipeline scoring, selection, transformation, guard, and assembly logic remain untouched.
- **Test impact:** P06/P07 are internal refactors with identical output — existing tests cover correctness. P08 needs new LRU eviction tests. P09 needs eviction + runner lifecycle tests.
- **Removed:** P03 (content-hash skip for installCursorHooks — P02's once-flag already guards; no bypass path exists), P04 (reuse SqliteStatusStore — trivial constructor with no I/O; negligible impact), P05 (reuse SqliteToolInvocationLogStore — same rationale as P04), P10 (esbuild single-file bundle for production — high risk from native addons, WASM, and path resolution issues; deferred for manual execution post-OSS release).
- **Planning hints:**
  - **P06:** The description mentions "mutable accumulation" as an option — this is NOT allowed. Project rules ban `let` and `.push()`. Use `Object.fromEntries(r.prunedFiles.map(f => [f.path, f.tier]))` — one expression, zero mutation. This is a one-line change in a pipeline file (`shared/src/pipeline/compilation-runner.ts`), so pipeline rules apply (max 60 lines/function, no Node APIs, no external imports).
  - **P07:** `serializeRepoMap` is a module-level free function (line 44), not a class method, but the hash also needs `this.stringHasher` (injected into `CompilationRunner`). The `WeakMap<RepoMap, string>` should be a private instance field on `CompilationRunner`, and a private helper method should check it before calling `serializeRepoMap` + `stringHasher.hash()`. `WeakMap` is a JS built-in (not a Node API), so pipeline rules allow it. `WatchingRepoMapSupplier.getRepoMap()` returns the same `RepoMap` reference until a file-system event invalidates the cache, so the `WeakMap` lookup will hit on every repeat compilation.
  - **P08:** Do NOT add the `lru-cache` package. Use a simple Map-based LRU: `Map` preserves insertion order, so delete-then-set moves an entry to the end; evict via `map.keys().next().value` when size exceeds the cap. This avoids a new dependency, ESLint restricted-import rules, and adapter wrapping. The file is `shared/src/adapters/caching-file-content-reader.ts` (29 lines) — the change is small.
  - **P09:** The `runnerCache` (in `mcp/src/server.ts`) should be a `Map<string, { runner: CompilationRunner; closeable: WatchingRepoMapSupplier }>`. In `getRunner`, instead of `createFullPipelineDeps`, call `createPipelineDeps` to get `PipelineDepsWithoutRepoMap`, then manually instantiate `WatchingRepoMapSupplier(new FileSystemRepoMapSupplier(...), new IgnoreAdapter())` and combine these into the `PipelineStepsDeps`. Store the `WatchingRepoMapSupplier` instance (which implements `Closeable`) in the cache alongside the runner. Implement a max size (e.g., 10 entries) using Map's insertion order for LRU eviction; when evicting, call `entry.closeable.close()`. In `registerShutdownHandler`, iterate `runnerCache` and call `entry.closeable.close()` on all remaining entries. This approach keeps all changes within `mcp/src/server.ts` and correctly manages the lifecycle of `fs.watch` instances without altering shared interfaces.
